Project 06

Geospatial ETL &
Geodatabase Automation

Designing and automating the data infrastructure that keeps a multi-project GIS environment consistent — from geodatabase schema design and CAD-to-GIS transformation standards, to Python and FME pipelines that synchronize datasets across disciplines without manual intervention.

DomainData Engineering

00
Context

A GIS environment supporting multiple concurrent projects across different disciplines — environmental, engineering, infrastructure — generates data from many sources: CAD design files from engineers, field surveys, provincial datasets, and project-specific geodatabases maintained by different teams.

Without consistent standards and automated synchronization, this data fragments quickly — leading to duplicate features, broken joins, incompatible projections, and reporting outputs that can't be trusted without manual reconciliation before every delivery.

01
Problem

Three interconnected problems drove this work — each compounding the others:

Data integrity chain (1) CAD files from engineering disciplines had no defined transformation standard into GIS — each conversion was ad hoc, producing inconsistent geometry and attributes.

(2) Multi-source datasets updated on different schedules were being manually re-synced before every report cycle — slow, error-prone, and undocumented.

(3) No organization-wide geodatabase schema existed — feature class naming, domain values, and subtype structures varied across projects, making cross-project queries unreliable.

The cumulative effect was that significant analyst time was spent on data reconciliation rather than data analysis — a structural problem that wouldn't be solved by working harder, only by building better systems.

02
Approach
Geodatabase Schema Design

An organization-wide geodatabase schema was designed from scratch — defining feature class structures, domains and subtypes to constrain attribute values, standardized field naming conventions, and version control procedures. The schema was documented and adopted as the baseline for all new projects.

CAD-to-GIS Transformation Standards

A formal CAD-to-GIS transformation workflow was developed to handle incoming engineering design files — defining layer mapping rules, coordinate system transformation parameters, geometry cleaning steps, and attribute population logic. This replaced ad hoc conversions with a repeatable, auditable process that preserved spatial accuracy and produced GIS-ready feature classes directly usable in analysis.

ETL Pipeline Automation

Geospatial ETL pipelines were built using Python and ArcGIS ModelBuilder to automate dataset synchronization across sources — pulling updates from provincial data feeds, engineering deliverables, and field collection systems, applying transformation logic, and loading into the target geodatabase with metadata intact. FME was used for more complex multi-format translations requiring conditional logic and cross-platform data alignment.

Cross-project data reliability

The geodatabase schema became the single structural standard across all projects — enabling cross-project queries, consistent reporting outputs, and onboarding of new datasets without custom transformation work each time.

The CAD transformation standard eliminated the most common source of geometry errors in the GIS environment — broken features from inconsistent layer mapping and missed coordinate system conversions from engineering deliverables.

Systems thinking over task execution The real value of this work wasn't any single pipeline — it was shifting the GIS environment from a reactive, per-project setup to a proactive, standards-driven infrastructure. Each new project plugged into existing schemas and pipelines rather than rebuilding from scratch.
