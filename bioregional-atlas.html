01Context
Capability demonstrated → Problem framing & spatial systems thinking

What problem does this solve?

Land-use planning, environmental review, and resource management decisions in BC depend on the ability to compare multiple sites across multiple dimensions simultaneously — terrain, ecology, resources, planning overlays, cultural context. Without a structured spatial system, that comparison is slow, inconsistent, and hard to defend.

This atlas was built to answer a specific organizational need: produce a consistent, replicable, multi-theme spatial reference for a large set of study sites — one that could be updated, extended, and queried without rebuilding the workflow each time.

Scale of the work

20+ sites × 15+ themes means 300+ individual map products, all derived from the same data library and layout logic. That scale is only achievable through systems design — not manual cartography.

02Data Architecture
Capability demonstrated → Understanding of BC's spatial data ecosystem

Organizing the data library

Before any maps could be produced, the underlying data needed a coherent, maintainable organizational structure. Datasets from BC Data Catalogue, TANTALIS, federal sources, and field surveys were catalogued into a thematic data library — the single source of truth for all atlas products.

This taxonomy did more than organize files. It defined the atlas structure, drove the cartographic template logic, and made it possible to add a new site or update a dataset without touching the rest of the system.

03Workflow
Capability demonstrated → GIS production pipeline design

End-to-end production system

The atlas was built on a reproducible pipeline — from raw data ingestion to exported map products. ArcGIS Pro Map Series (Data Driven Pages) drove cartographic output: once the template and symbology were set, the same layout generated maps for every site automatically.

Python scripting handled batch processing, projection management, and geodatabase operations. The pipeline was designed so that adding a new site required only updating the data library — not rebuilding the workflow.

04LiDAR Automation
Capability demonstrated → Computational GIS & automated terrain analysis

Automating the hydrology theme

The hydrology and terrain theme required surface-derived products (DTM, DSM, CHM, slope, aspect, hillshade) and hydrological analysis outputs (stream networks, watersheds, pour points) — all generated from raw LiDAR point clouds.

An R script using lidR handled point cloud normalization and surface derivation. A Python script ran hydrological analysis and fed results into a folium map for inline QA visualization — so outputs could be verified without opening GIS software between runs.

Why this matters at scale

Manual terrain and hydrology analysis for 20+ sites would have taken weeks. Automating the pipeline reduced per-site processing to a single script execution — and the embedded visual QA caught classification errors before they propagated into the atlas.
